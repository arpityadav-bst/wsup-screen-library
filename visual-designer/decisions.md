| Gate 5 strengthened — subjective "did visual language change?" replaced with mechanical file-presence check | **Failure mode caught at end-of-S27:** despite Gate 5 saying "SAME EDIT," drift accumulated across 4+ component edits today. ChatBar lost its bulb but `ChatBarSection` still showed it. ChatHeader got the 3-dot menu wired but the section didn't note it. MenuPopover became a shared primitive but lived only as anatomy notes in another section, never as its own Components-tab entry. The audit caught it after the designer pushed — that's a backstop firing, not the gate working. **Three failure causes: (1)** subjective triage ("if visual language changed") gave a loophole to skip the gate when I rationalized "I'm just removing an element"; **(2)** I conflated "mentioned in someone else's anatomy" with "has its own section"; **(3)** I asked the forward question (new thing → new section) but never the reverse (modified existing thing → which sections demonstrate it?). **Gate 5 rewritten** to be mechanical: when editing a component file, grep for any section that imports it OR demonstrates its visual inline. If found, that section is part of THIS edit's scope. No interpretation. New primitives in `ui/` aren't done until they have their own section + ComponentsTab import + NAV registration. The strengthened rule lives in QUALITY-GATES.md so future sessions read it at session bootstrap. **Why this matters:** style-guide drift is the failure mode that splits the dev handoff from the live demo — devs read one truth, ship another. Gate 5 is the only thing standing between code and that split. Subjectivity in the gate's trigger makes it leak |
| Pill bulb-to-text spacing: settled at `gap-[6px]` after gap-xs (8px, too loose) → gap-xxs (4px, too tight) | **Designer-caught:** my last swing went too far the other direction. WSUP spacing scale jumps from xxs (4px) directly to xs (8px) — there's no native 6px token. Per Gate 1's accepted-exceptions clause, "near-match micro-adjustments where forcing to nearest token would break the design" explicitly lists `gap-[6px]` as legitimate. Used it. **Generalization:** when a 4px → 8px scale-step is the wrong size of jump for the intended visual rhythm, the arbitrary `[6px]` is the right answer — not a new token (one-off doesn't earn one), not forcing to the nearest token (breaks the design) |
| Toast width: switched from max-w (auto-shrinks to content) to explicit w-[calc(100vw-32px)] on mobile | **Designer-caught:** mobile toast was wrapping into 3 lines with significant gutter on left/right — wasted space made the wrapping LOOK heavy when it didn't need to. **Root cause:** `max-w-[calc(100vw-32px)]` only constrains maximum; actual width follows content. Short messages auto-shrink → narrow toast → forced wrap. **Fix:** explicit `w-[calc(100vw-32px)]` on mobile fills full viewport minus 16px each side. Desktop stays `md:w-auto md:max-w-[520px]` so it auto-sizes for one-line messages |
| Toast position: top on BOTH mobile and desktop (designer reverted my bottom-mobile placement) | **Designer reverted my call:** the previous turn moved mobile toast to bottom (thumb-reach reasoning). Designer redirected: keep top on mobile too. Likely reasoning (worth capturing): the toast is feedback for an action originating in the chat 3-dot menu at the TOP-RIGHT. Feedback at the TOP maintains spatial continuity with the action source — bottom-positioning would disconnect the feedback from the trigger ("I tapped at the top, why is the response at the bottom?"). Spatial continuity beats thumb-reach when the originating action wasn't itself in the thumb zone. **Implementation simplified:** single `top-[80px]` position, no viewport-conditional toggle. **Lesson — taste rule deleted, not replaced:** I added a "top on desktop, bottom on mobile" rule last turn based on a generic-feeling Material Design instinct. Designer's reversal proves it's not the right rule for *this* product. Removed from taste.md rather than replaced — better to have no rule than a wrong one |

Flipped: `fixed top-0 z-[79]`, gradient direction `to bottom` (dark at top, transparent at bottom). 220px tall, covers the chat header (56px) and the toast band (top-[80px] + ~50px) with breathing room below. Stacks with the chat header's existing `bg-gradient-to-b from-black to-transparent` (mobile only) for guaranteed contrast against the character image. Desktop has no scrim — plain dark page-bg already provides enough contrast. **Generalization preserved (taste.md):** "scrim what's BEHIND the surface, don't make the surface itself heavier" — that rule still holds; only the location of the scrim changed |
| SuggestedReplies collapsed pill: tightened bulb-to-text gap, added right chevron (matches model picker pattern) | **Designer-caught:** the previous square-wrapper approach gave the bulb uniform 8px breathing on all sides (good) but then `gap-xs` (8px) ADDED to the square's 8px right padding meant 16px total visible distance from bulb to text. Felt loose. **Fix:** dropped the `w-8 h-8` square wrapper. Bulb sits directly inside the pill at `pl-xs` (8px) + `gap-xxs` (4px) before text. The pill's `h-8` auto-centers the 16px bulb vertically — `(32-16)/2 = 8px` top/bottom — naturally matching the 8px left padding. Bulb still has uniform 8px breathing on left/top/bottom; bulb-to-text drops from 16px → 4px. **Square wrapper rule from previous session is now narrowed:** use it when the icon's container needs to be perceptibly distinct (e.g., the inner anchor circle pattern); when there's no such need, direct padding is tighter and equally balanced. **Added a chevron** (4×8 SVG, strokeOpacity 0.5) at the right end of the pill — same anatomy as the ChatBar model picker. Visual signal that the pill expands. Subtle, not shouty |
| Toast: dropped "(⋮)" suffix from disable message, widened on desktop (520px max-w + md:whitespace-nowrap), wider mobile bound (calc(100vw-32px)) | **Designer-caught three issues:** (1) The "(⋮)" character at the end of "Turn back on from the chat menu (⋮)" was redundant — the kebab glyph isn't universal enough to add clarity, and the user immediately reads the text "chat menu" without needing visual emphasis. Just text is fine. (2) On desktop, the toast wrapped to 2 lines because `max-w-[360px]` was too narrow for the message. Should be a one-liner on desktop. (3) On mobile, the toast wrapped heavily because the same 360px max collided with the message length AND the disable message had been padded with extra characters. **Fixes:** (a) Removed "(⋮)" from message — "Auto-suggestions off. Turn back on from the chat menu." (b) Widened toast: `max-w-[calc(100vw-32px)] md:max-w-[520px]` — desktop fits one line at 520px, mobile fills available width minus standard 16px-each-side margins. (c) Added `md:whitespace-nowrap` to the inner `<p>` so desktop never wraps even if text grows slightly later | **Designer-caught:** the collapsed pill's left padding read as larger than its top/bottom padding, even though the explicit `pl-xs` (4px) matched the vertical breathing around the inner 24×24 circle. **Root cause:** the inner circle's distinct `bg-white-10` against the pill's `bg-black-60` made the eye treat the circle as a separate element with its OWN padding-from-pill-edge — a circle-on-pill effect that visually doubled the perceived left margin. **Fix:** removed the inner circle entirely. Bulb icon (bumped 14→16px) now sits inside a `w-8 h-8` square wrapper centered with `flex items-center justify-center`. Result: bulb has uniform 8px breathing on all four sides — `(32-16)/2`. The square's left edge curves into the pill's rounded cap naturally, so the bulb visually sits inside the cap, not floating beside it. **Generalization (added to taste.md):** when an icon needs uniform breathing inside a pill/badge, give it a square wrapper sized to the container's height — don't try to balance left padding against vertical padding by trial and error |
| SuggestedReplies expanded: × moves to LEFT (same position pill occupied), bulb-off stays RIGHT | **Designer-caught two related issues:** (1) Original layout had × and bulb-off adjacent on the right → real misclick risk between two icon buttons of similar shape. | Bulb-off icon: replaced manual strikethrough overlay with single integrated SVG (Lucide-style lightbulb-off) | **Designer-caught:** the previous bulb-off was an `<img>` of `icon-bulb.svg` with a CSS-positioned `<span>` painted diagonally across it. The two layers didn't share the same anti-aliasing, line-weight, or geometry — so the strikethrough read as "applied to the icon" rather than "part of the icon." Jarring visually. **Fix:** wrote a proper single SVG path that draws a bulb shape AND the diagonal slash within the same `<svg>` viewBox, with consistent `stroke-width: 1.8` and `stroke-linecap: round` across all paths. The slash now belongs to the icon's design rather than overlaying it. Same semantic ("the suggestion bulb is off"), much cleaner execution. **Lesson:** if an icon-with-modifier (off, mute, blocked) is needed, source or draw a single integrated SVG — don't compose it from two layered elements at runtime |
| MenuPopover restored to render TWO surface styles — sheet-style on mobile, popover-style on desktop — with shared items[] data (designer-caught my over-correction) | **What I got wrong:** in the previous turn the designer said "shouldn't this be same everywhere?" referring to chat menu vs profile menu spacing differing. I interpreted that as "use ONE visual style across all menus, mobile and desktop alike." Wrong. The designer actually wanted: keep the profile MenuSheet's tap-friendly mobile sheet (left-aligned rows, title label, dividers, Cancel) AND componentize THAT for all mobile menus. Desktop stays popover-style. They wanted ONE shared component, but TWO viewport-appropriate surfaces inside it. **What sheet-style gives that popover-style doesn't on mobile:** tap targets (40px+ rows vs 28px), readable text size (`text-base font-medium` vs `text-sm font-semibold`), categorial title that orients the user ("CHARACTER", "Account", "Chat"), Cancel button as a confirmed-exit affordance (mobile users may not realize backdrop-tap dismisses). Internally renders `<MenuItem mode="sheet" />` rows for mobile body and `<MenuItem mode="popover" />` rows for desktop popover. Same `items` array drives both — viewport picks the surface, not the consumer. **Per-menu code is now ~15-25 lines** of pure config — `ChatHeaderMenu`, `MenuSheet`, `PublicMenuSheet` are all thin wrappers. **Wrong taste rule corrected:** "One pattern wins app-wide" was too aggressive. Replaced with the right framing: same viewport-role gets the same pattern, but viewport itself can legitimately have different patterns when mobile/desktop affordances genuinely differ |

| First-click bug fixed: ChatBar's outside-click handler now respects a shared `containerRef` covering the suggestions surface | **Bug:** First tap on the suggestions pill caused ChatBar to collapse but pill DIDN'T expand — user had to tap twice. **Root cause:** ChatBar's document-level `mousedown` listener fired on the pill click (pill is outside ChatBar's wrapper) → `setIsActive(false)` → ChatBar shrunk vertically (model picker row 2 disappeared) BETWEEN mousedown and mouseup → user's mouseup landed on a different DOM element than mousedown → `onClick` (which requires both on same element) didn't fire on the pill.
